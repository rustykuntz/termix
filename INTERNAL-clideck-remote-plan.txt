CliDeck Remote: Internal Product + Legal Note

Why we are doing this
We want one seamless CliDeck user experience in the desktop app, so users can discover and activate Mobile Remote from the same UI.
At the same time, the actual remote/mobile implementation stays separate and proprietary, outside the MIT open-source repo.

Product split
1. OSS desktop app
   Repo: /Users/rusty/Projects/termix
   Role: local dashboard, session management, plugin system, Mobile Remote entry point UI
   License posture: MIT
   Important: no relay, no mobile app, no remote protocol implementation, no hosted-service logic in this repo

2. Remote connector
   Repo: /Users/rusty/Projects/clideck-remote
   Package: clideck-remote
   Distribution: proprietary npm package
   Role: local bridge/companion that handles pairing, plan state, QR generation, and relay communication

3. Hosted remote service
   Repo: /Users/rusty/Projects/clideck-remote
   Role: Cloudflare relay + mobile web app + service-side logic
   License posture: private/proprietary

Business model
- Free: 1 concurrent mobile-controlled session
  Rule: if the user activates another mobile session, the previous mobile-controlled session becomes dormant
- Pro: $7/month, unlimited mobile concurrent sessions
- No Teams plan for now

Desktop UX
- The mobile icon stays in the OSS desktop app
- First click:
  - if clideck-remote is not installed, show an intro screen explaining what CliDeck Mobile Remote is
  - show one main button: Add to CliDeck
  - do not show pricing on this first screen
- After install, or if already installed:
  - go straight to the QR/mobile URL screen
  - show current plan state
  - default to Free if no paid state is known
  - show Pro if Stripe/Link says the user is Pro
  - if Free, offer Upgrade to Pro
  - if Pro, show Pro active

Boundary between OSS and paid product
- OSS app may show the icon, intro screen, QR screen container, and plan UI
- OSS app may detect whether the connector exists
- OSS app must not contain the remote implementation itself
- clideck-remote owns:
  - pairing
  - entitlement/plan check
  - QR/mobile URL generation
  - relay communication
  - mobile control logic

Why this split is legally important
- The public repo stays genuinely MIT and local-only
- The paid remote service is not accidentally open-sourced
- Users can still discover the paid product from inside CliDeck without putting the proprietary implementation in GitHub
- Hosted-service legal docs should live with the private repo and/or the website, not in the OSS repo

Practical rule
If a file is required to operate the paid remote/mobile service, it belongs in clideck-remote, not in termix.
