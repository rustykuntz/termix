Terminal UI Workplan

Goal
- Build a local, PTY-based terminal dashboard inspired by WhatsApp Web for managing many CLI agent sessions with better scanability, launch controls, and session organization.

Current State

Implemented foundation
- PTY sessions run through `node-pty` and render in the browser via xterm.js.
- The `node-pty` macOS `spawn-helper` permission issue is handled automatically at startup.
- Settings persist to a local `config.json` file in the project root.
- Config schema includes: `defaultPath`, `commands`, `profiles`, `prompts`, `defaultProfile`, `confirmClose`.
- Agent presets defined in `agent-presets.json` (Claude Code, Codex, Gemini CLI, OpenCode, Shell).
- Server sends presets on WebSocket connect (`type: 'presets'`).
- Session creation uses an inline creator card in the sidebar panel:
  - Random fun session name placeholders (Color + Animal combos, e.g. "Blue Panda").
  - Agent preset buttons with icons rendered from presets message.
  - MRU ordering: last-used agent floats to top, Shell always last.
  - Clicking an agent auto-creates the matching command in config if needed, then spawns the session.
- Left navigation rail with 4 icons: Sessions, Projects, Prompt Library, Settings.
  - Rail acts as a unified menu — clicking any icon switches context.
  - Settings opens as an overlay in main; panel items switch sidebar content.
- Profiles / theme system:
  - 10 built-in terminal themes (Default, Dracula, Nord, Catppuccin Mocha, Solarized Dark, One Dark, Monokai, Tokyo Night, GitHub Dark, Gruvbox Dark).
  - Per-session profile switching via context menu.
  - Progressive settings UI: single profile = "Appearance" (just theme + accent), multiple = "Terminal Styles" with cards, stars, and MRU.
  - Custom Tailwind-styled theme dropdown with color preview strips (replaces native selects).
- Settings overlay with two-panel layout: category nav sidebar (General, CLI Agents, Appearance) + content panels.
- Close confirmation modal (toggleable via settings).
- Folder picker modal with directory browsing.
- Tailwind CSS v3 compiled once to static file (`npm run build:css`).
- Session resume infrastructure (backend): sessions.json persistence, resumable session list, resume command support.

Current code structure
- Backend:
  - `server.js` — HTTP/WebSocket bootstrap
  - `config.js` — config load/save with migration
  - `sessions.js` — PTY/session lifecycle, resume, persistence
  - `handlers.js` — WebSocket message dispatch (sends config, themes, presets, sessions, resumable on connect)
  - `utils.js` — shared backend helpers
  - `themes.js` — 10 built-in terminal theme presets
  - `stats.js` — TEMPORARY process stats (CPU, memory, network per session via process group)
  - `agent-presets.json` — CLI agent definitions with outputMarker field
- Frontend (ES modules in `public/js/`):
  - `app.js` — app bootstrap, WebSocket message routing, stats overlay + working/idle detection
  - `state.js` — shared client state (ws, terms, cfg, themes, presets, resumable)
  - `terminals.js` — terminal add/remove/select/rename, session context menu, buffer reading for agent output
  - `settings.js` — settings panel rendering, custom theme dropdown, save logic
  - `creator.js` — inline session creation card with agent preset buttons
  - `nav.js` — left navigation rail panel switching
  - `profiles.js` — profile/theme resolution helpers
  - `confirm.js` — close confirmation modal
  - `folder-picker.js` — directory browser modal
  - `utils.js` — small frontend helpers (esc, debounce, agentIcon)
- Styling:
  - `src/input.css` — Tailwind source with custom `.term-wrap` rules
  - `public/tailwind.css` — compiled Tailwind output
  - `tailwind.config.js` — Tailwind v3 config

Product Direction
- Keep the PTY architecture.
- Keep the code slim and local-first.
- Prefer simple, explicit data models over smart-but-fragile abstractions.
- Continue moving toward a WhatsApp Web-style management experience.

- Close confirmation modal (toggleable)

- Left navigation rail (Sessions, Projects, Prompt Library, Settings)

- Custom Tailwind-styled dropdowns (theme picker with color preview strips)
- Tailwind CSS v3 integration
- Static file path traversal protection
- Session resume backend (persistence, resumable list, resume commands)
- WhatsApp-style session list layout:
  - Agent icon (circular avatar per CLI tool)
  - Session title (bold, user-defined)
  - Last activity timestamp
  - Status indicator (working spinner / idle zzZ) with smooth fade transition
  - One-line preview of last agent output (via xterm.js buffer reading)
  - Context menu (down-arrow on hover) with Profile and Delete options
- Session status: working / idle detection:
  - Network-based: output rate >= 800B/s + burst increasing + no user typing = working
  - Idle: output rate < 800B/s + burst not increasing, sustained ~1.5s
  - Per-session process stats (CPU, memory, network) via process group tracking
  - Stats overlay toggle in Settings > Appearance > Monitoring
- Agent output detection via `outputMarker` field per agent preset:
  - Each agent declares its output line marker (⏺ Claude, • Codex, ✦ Gemini, │ OpenCode)
  - Buffer reader scans xterm.js screen bottom-up for the marker to extract last agent message
  - Works across themes since it reads character content, not colors

Next Phase Priorities

1. Folder Picker Polish
- Keep the browse-style directory picker as the primary path selection flow.
- Improve the remaining UX gaps:
- hidden-directory access when needed
- clearer disabled/error states
- better validation feedback when a path cannot be resolved

2. Projects
- Add a lightweight project layer above sessions.
- A project is a grouping container for multiple sessions.
- The hierarchy should become:
- App
- Project
- Session

Recommended v1 scope
- project id
- project name
- optional icon or accent color
- active project selection
- filter the session list by the selected project
- fallback project such as `General` or `No Project`

Session behavior
- Every session should belong to a project.
- When creating a new session, allow associating it with a project.
- If the user does not choose one, assign it to the default fallback project.
- Switching projects should scope the visible session list to that project.

3. Session Metadata Layer
- Add per-session metadata needed for richer list behavior.
- Track:
- `createdAt`
- `lastInputAt`
- `lastOutputAt`
- `lastViewedAt`
- `lastActivityAt`
- `lastPreviewText`
- command id/label
- command type / icon id
- project id
- cwd
- This is the foundation for timestamps, previews, search, and unread.

4. Search
- Add a search bar above the session list.
- Recommended v1 search scope:
- session name
- last preview text
- optional rolling terminal buffer

5. Filters: All / Unread
- Add two pill filters:
- `All`
- `Unread`
- Unread should derive from output activity since last view/read event.

6. Prompt Library
- Add a prompt library section in Settings.
- Store reusable prompts/templates in config.
- v1 interaction:
- open prompt picker
- insert into active PTY
- optional mode to auto-send with newline

7. Resume UI
- Show resumable sessions in sidebar with resume button.
- Wire up to existing backend resume infrastructure.

8. Session Creation Enhancements
- Still TODO: per-command default profile, per-command default working directory override.

Design Notes
- Settings UI uses Tailwind CSS with custom styled dropdowns (no native selects).
- Progressive disclosure: simple Appearance section when one profile, full Terminal Styles cards when multiple.
- The refactor is a good baseline; further feature work should preserve module boundaries.
- New features should be implemented on top of explicit session metadata, not inferred from DOM state.

Known Review Note
- Keep the static file serving path traversal protection intact as the server evolves.

Open Questions
- Should `Projects` be its own rail view with a project list, or should it directly switch the session column into grouped project mode?
- Should the fallback project be called `General`, `Inbox`, or `No Project`?
- Should profiles stay purely visual in v1, or also include command/path defaults?
- Should resumable sessions auto-restore on startup, or should restore always be explicit?
- How should we represent non-resumable historical sessions after restart: archive, closed entry, or hide them?
- Should unread clear on focus, click, or only after explicit interaction?
- Should prompt library default to paste-only, or paste-and-send?
- Should the folder picker allow hidden directories in an advanced mode?
